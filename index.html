类加载机制

首先编写的源码文件，也就是后缀为.java文件。会经过源码编译器，编译成.class文件。再通过类加载器classLoader加载进JVM虚拟机。

JVM类加载机制主要分为加载、连接、初始化等阶段。

加载

加载过程主要是：
（1）通过类的全限定类名获取此类的二进制流。
（2）根据二进制流，在JVM的方法区保存类的信息。
（3）在堆中创建class对象，指向方法区类元信息。

连接

校验

确保加载的二进制流，符合JVM数据要求。

准备

为静态变量开辟内存空间并初始化

解析

将类中的符号引用转换为直接引用。符号引用类似占位符，没有实际意义。

初始化

为类中的静态变量、静态代码进行初始化操作

类加载器

加载原则 双亲委派

当一个类加载器收到类加载请求时，会将请求转给它的上一级类加载器。如果最上级的类加载器不加载，才会由下面的加载器进行加载。
双亲委派机制保证了类加载的唯一性。

ClassLoader的loadClass方法提供了双亲委派加载的实现。

JVM运行时数据结构

方法区

方法区是所有线程共享的。主要存储类元信息、常量、静态变量。方法区中有常量池。

堆

存储实例对象、数组数据。所有线程共享。

虚拟机栈

虚拟机栈线程私有，线程每次调用方法，就会有一个该方法的栈帧入虚拟机栈，在方法调用完成后出栈。

一个栈帧由局部变量表、操作数栈、动态链接、方法返回地址组成。
局部变量表：对应了方法的入参，还有方法体中定义的变量。
操作数栈：对于数据的操作，都是在操作数栈中完成。
动态连接: 指向所调用方法的引用。

程序计数器

记录线程上次执行指令的位置，用于线程切换后的恢复。

sychorzied锁升级过程

锁升级设计的目的主要为在没有多线程竞争的场景下，减少加锁带来的资源消耗。

java对象MarkWord

首先分配在堆中的java对象由对象头、实例数据、对齐填充组成。

在markword中存储锁相关的信息，是实现锁升级的关键。

无锁->偏向锁

初始状态是无锁状态。当第一个线程进入同步代码块，markword中的锁标志会更新为01，并通过cas操作将当前线程存入markword中。当下次同一个线程再次进入同步代码块，会比较markword中的线程id是否一致，若一致就不需要再次加锁。

若是在必定发生多线程竞争的场景下，用参数-xx:-UseBiasedLocking来禁止偏向锁提示性能。

偏向锁->轻量级锁

1、首先在当前线程的栈帧中创建锁记录LockRecord。
2、将markword拷贝到锁记录中，也就是加了displaced的位置。

3、将锁记录中的owner指向对象头。
4、将对象头指向锁记录中的markword复制

多线程(二) volatile

硬件层面并发效率问题

CPU并发执行虽然能提升运行效率但受限于CPU、内存、存储设备之间的速度差异。因此通过引入高速缓存来为CPU、内存做缓冲，CPU从高速缓存中获取计算数据，完成计算后数据再由高速缓存更新到内存。

高速缓存的引入带来新的问题：缓存一致性。多个cpu都有缓存，当它们都去缓存共享内存中同一片数据就可能发生不一致的情况（cpuA更新缓存并同步至共享内存，cpuB依然使用之前的脏数据）。硬件层面设计了缓存一致性协议如MESI（通过改变cpu本身缓存数据的状态，并监控其他cpu的缓存状态）。

java内存模型（JMM）

java内存模型规定所有的变量都在主内存中，每个线程都有各自的工作内存。线程操作变量首先要将主内存的数据复制到工作内存中来进行计算。线程间数据值的传递也要通过主内存。


内存模型定义了原子性操作lock、unlock、read、load、use、assign、store、write。并规定了内存访问安全的场景。

volatile

volatile是轻量级的同步机制，它解决了线程安全的可见性、有序性问题，并没有解决原子性。

可见性

修改volatile变量时，底层指令会加上lock指令。所以变量的值会立即更新至主内存，其他线程工作内存中保存的脏数据失效。

有序性

根据as-if-serial（单线程下执行结果结果相同）
cpu在不影响程序最终结果（不存在数据依赖）的前提下，可能会将指令重排序。
int x = 1;
int y = 2;
int z = x;
虽然在代码中x的赋值在y之前，但第三行紧接着又用到了x。处理器就可能调换赋值x、y的顺序，以此来提示效率。

重排序在多线程下的问题

线程A执行writer(),线程B执行reader()。操作1、操作2不存在依赖，就可能发生重排序导致操作2先执行，那么线程B实际读到的a = 0。

volatile修饰的变量会加入内存屏障来保障不会进行指令重排序。

算法-动态规划（斐波那契_青蛙跳台阶_01背包）

动态规划的基本思想是通过子问题的结果推导出问题结果。先把子问题结果都保存起来，要求问题结果时，从保存的结果里找到之前的结果组合成新问题的结果。同时遵循自左向右，自下而上。

斐波那契


F（n）= F(n-1)+F(n-2)是问题结果的推导。n时的结果是n-1的结果加n-2的结果。用数组来保存之前的结果，要求N时，就去数组里拿。

这题也能用递归求解，但耗时过长。每次递归都要重新计算结果。

青蛙跳台阶

要求n级台阶跳的次数。要知道青蛙最后一次跳台阶，要么跳一格要么跳二格。当跳一格时，那他已经跳了n-1格台阶，也就是有f(n-1)种方法跳到n-1级台阶。当跳两格时，那他已经跳了n-2格同理就是f(n-2)。进而发f(n)=f(n-2)+f(n-1)

01背包

给定 n 个物品和一个容量为 W 的背包，物品 i 的重量是 wi，其价值为 vi 。应该如何选择装入背包的物品，使得装入背包中的物品的总价值最大？
（输出给定n和W时的最大总价值）

例：
有如下5个物品，小明的书包最多只能装下8公斤的物品，小明特别贪心，思考怎么选择使自己书包能装下并且得到的价值最大。

物品1:6公斤　　 价值48元
物品2:1公斤 　　 价值7元
物品3:5公斤 　　 价值40元
物品4:2公斤 　　 价值12元
物品5:1公斤 　　 价值8元


建立二维表，行代表背包容量。列代表可以放入的物品。遵循自左向右自下而上。从左下角开始填表，当背包容量为1kg和有物品5的情况下，最大价值就是8元。当背包容量为1kg和有物品4、物品5的情况下，最大价值为12元。
逐步将问题抽象为，第n件物品放还是不放的问题。以上图表格为例。现在要求行为2，列为3的值。也就是背包容量为3kg,可放物品有2、3、4、5。

如果物品2不放，就是求背包容量为3kg,可放物品有3、4、5的值，也就是标黄格子的值。
如果物品2放，要把物品2的1kg空出来，背包也就只有2kg,可放物品为3、4、5。也就是红格子所代表的值。拿红格子的值加要放入背包的物品2的值就是这个子问题的解。

最后比较这两个子问题，哪个大就选哪个。

多线程（一）

进程与线程

线程是进程的一部分，线程本身不会单独存在。进程是操作系统资源分配的基本单位，线程则是资源分配的最小单位。

线程存在的意义在于提升CPU的运行效率。一个CPU单位时间只能执行一个任务，假如这个任务执行过程中阻塞，那CPU资源就被闲置。因此加入线程，在JVM层面每个线程拥有自己的虚拟机栈与程序计数器。多核CPU下，CPU执行中线程可以被挂起，根据线程的程序计数器又能从上次执行到的地方继续执行。

线程状态

操作系统进程五态

新建状态NEW:进程被创建时所处的状态。

就绪状态READY:进程已经具备了可执行条件，等待CPU调度。

运行状态Running:进程占用CPU正在执行。

阻塞状态Blocked:进程受到通信或同步操作影响，不得不放弃CPU使用。

终止状态Terminated:进程运行结束或出现严重错误。收回占用资源。

就绪-》运行：CPU分配给某个就绪的进程。

运行-》阻塞：正在运行中的进程需要等待某些事件的发生。

运行-》就绪：CPU分配的时间耗尽或其他优先级更高的进程插队

java线程状态

NEW：还未调用start()的线程处于NEW状态。

RUNABLE: 包含了处于就绪Ready和运行Running

WAITING、TIMED_WAITING只是加入了超时时间。

线程创建

有三种创建线程的方式：实现Runable接口、继承Thread类、实现Callable

callable可以拿到线程执行的结果

线程通知机制

Object类是所有类的父类，它拥有wait()、notify()等方法。

要先使用这些方法，首先必须要拿到该对象的监视器锁（在synchronized修饰的代码块中）。

wait():线程挂起，并释放对象的监视器锁。加入锁对象的等待队列。

notify():随机唤醒一个该对象等待队列中的线程，使之进入同步队列去争抢监视器锁。notify并不会释放锁

threadA先启动，之后主线程谁100ms，确保线程A先拿到对象o的监视器锁。线程A输出后，调用notify()唤醒处于等待队列的线程，此时并没又唤醒任何线程，然后wait()阻塞。此后，主线程睡眠结束，线程B开始执行。线程B输出后，执行notify()将线程A从等待队列唤醒进入同步队列，然后线程Bwait()释放锁，让线程A执行。

threadA先启动，之后主线程谁100ms，确保线程A先拿到对象o的监视器锁。线程A输出后，调用notify()唤醒处于等待队列的线程，此时并没又唤醒任何线程，然后wait()阻塞。此后，主线程睡眠结束，线程B开始执行。线程B输出后，执行notify()将线程A从等待队列唤醒进入同步队列，然后线程Bwait()释放锁，让线程A执行。

wait()、notify()多线程交替输出

在线程获取到锁对象后，执行锁对象的wait()。当前线程将释放锁并加入锁对象的等待队列，其他线程调用锁对象的notify()、notifyAll(),则将之前加到等待队列的线程改变到锁对象的就绪队列中。等到CPU轮转到这个线程，就继续执行输出。
